# Lintel — Feasibility Research Appendix

*Generated 2026-07-06 from an orchestrated 10-dimension investigation (31 agents, ~1.37M tokens). Each dimension's single riskiest 'critical claim' was adversarially double-verified (docs lens + real-world/prior-art lens). This appendix preserves the per-dimension detail, verdicts, and sources behind [the design](../plans/2026-07-06-lintel-design.md).*

## Overall feasibility

FEASIBLE for the fully-interactive version, with two genuinely novel/unproven areas that must be de-risked by early spikes. The read-and-trigger core is de-risked by shipping prior art: Many Tricks' Menuwhere has commercially read the frontmost app's full menu tree over AX and executed items (incl. submenus and background apps) for years, and the MIT-licensed right-click-menubar reproduces the exact code path (AXUIElementCreateApplication -> kAXMenuBarAttribute -> recursive kAXChildren -> AXUIElementPerformAction(kAXPressAction)). The non-activating clickable overlay (NSPanel + NonactivatingPanel) is confirmed high-confidence, and drawing OVER the system menu bar via window levels >24 is confirmed high-confidence (shifting it aside is infeasible — abandon that sub-goal). Every required API has a public objc2 binding (objc2-application-services 0.3.2 gained the full AX C surface after issue #624); no private API is required except the optional AXFullScreen fullscreen probe, for which a public NSScreen.frame-comparison fallback exists.

BIGGEST THREAT: lazy/dynamic submenu population. The verifiers narrowed the original claim — Window, Font, and Open Recent are actually AppKit-managed persistent NSMenuItems that ARE AX-readable without opening, and a passive AXChildren read often triggers -[NSMenu update]/menuNeedsUpdate: without any visible display. But a real residual class remains: menus built at tracking-start in NSMenuDelegate menuWillOpen:, and non-native toolkits (Electron/Chromium, Qt, Java/JUCE) whose menu items do not exist in the AX tree until the menu is physically opened. For those, the only cross-process lever (AXShowMenu/AXPress on the parent) visibly drops the real menu, which fights Lintel's unobtrusive-mirror goal and can yield a transient flash and initially-stale enabled state. This is bounded (a minority of menus) and mitigable (lazy on-hover, cache), but it is the one thing that cannot be made fully invisible for all apps. Two secondary threats: (a) whether NSGlassEffectView / NSVisualEffectView behind-window blur actually samples a DIFFERENT app's live window sitting under a non-activating overlay (unverified — prototype early, fall back to NSVisualEffectView); and (b) keeping the bar pixel-glued during live drags and window-server-driven moves (Stage Manager/Spaces/tiling), which requires a reconciliation poll on top of AX notifications.

## Headline risks

- LAZY / DYNAMIC SUBMENU POPULATION (severity: high). menuWillOpen:-built menus and non-native (Electron/Qt/Java) menus have no AXMenuItem children in the AX tree until physically opened; the only cross-process force-open (AXShowMenu/AXPress the parent) visibly drops the real menu and can flash. MITIGATION: eagerly read only statically-defined top-level + first-level items (a passive AXChildren read triggers -[NSMenu update] for most native menus without a visible open); mark a submenu 'dynamic' when its AXMenu reports zero children but the parent is submenu-capable; populate those lazily on hover by AXShowMenu-ing the real submenu, poll AXChildren until non-empty (bounded by AXUIElementSetMessagingTimeout), read, then dismiss; cache per app and refresh via AXMenuOpened/AXMenuClosed observers; accept a brief flash for that minority and feature-detect apps that expose no usable AXMenuBar.
- REAL-TIME GEOMETRY TRACKING (severity: high). kAXWindowMoved/kAXWindowResized are not contractually guaranteed, are throttled/coalesced (native apps lag during fast drags), and are frequently NOT emitted for window-server-driven moves (Stage Manager, Spaces, tiling). MITIGATION: observe AXWindowMoved/AXWindowResized/AXWindowMiniaturized/AXUIElementDestroyed on the focused-window element as the primary signal; add a low-rate reconciliation timer that re-reads kAXPosition/kAXSize (and re-derives the owning screen) to catch missed programmatic moves; hide/detach the bar during rapid motion and re-pin on settle; debounce. Note the Electron/Java top-level window IS a native NSWindow that posts geometry notifications — only its content AX tree is lazy, which Lintel does not need for placement.
- TCC ACCESSIBILITY GRANT PERSISTENCE ACROSS REBUILDS (severity: high, dev-workflow). An ad-hoc/unsigned build's cdhash changes every rebuild, so its stored csreq no longer matches and the grant silently resets (the classic yabai/skhd pain). MITIGATION: from day one, ship a real .app bundle with a STABLE CFBundleIdentifier and sign every build with a REUSED stable code-signing identity — a persistent self-signed cert is sufficient for TCC persistence (a Developer ID cert + notarization is only needed for distribution to other Macs, not local use); wire `codesign` into the build after cargo-bundle; use `tccutil reset Accessibility <bundle-id>` to clear stale entries. Prefer a bundle (client_type=0) over a bare binary because macOS 26 System Settings does not render path-based TCC entries.
- FIRST-CLICK NOT DELIVERED TO CUSTOM VIEWS (severity: medium). NSView.acceptsFirstMouse defaults to false, so while Lintel is inactive (the normal case) the first click on a custom-drawn menu item is consumed only to make the panel key rather than firing the item. Also, setActivationPolicy(.accessory) does NOT by itself prevent becoming frontmost — the NonactivatingPanel style mask does. MITIGATION: override acceptsFirstMouse(for:) -> true on the bar's interactive views via define_class!; rely on the NonactivatingPanel mask (reinforced by overriding canBecomeMainWindow, and canBecomeKeyWindow only if a search field needs keyboard); use makeKeyAndOrderFront/orderFront to show the panel (safe, does not activate) and NEVER call NSApplication.activate(ignoringOtherApps:).
- BEHIND-WINDOW GLASS SAMPLING OVER A FOREIGN LIVE WINDOW IS UNVERIFIED (severity: medium). It is not documented whether NSGlassEffectView / NSVisualEffectView(.behindWindow) blur updates live over a DIFFERENT app's window sitting beneath a non-activating overlay. MITIGATION: build a styling spike in Phase 1; if glass sampling misbehaves over foreign windows, fall back to NSVisualEffectView (a decade of proven behind-window behavior), and set NSVisualEffectState::Active explicitly since the non-activating panel never becomes key (default followsWindowActiveState would render it dimmed).
- OVER-THE-MENU-BAR LAYERING ON macOS 26 (severity: medium). A bar at only NSStatusWindowLevel (25) is overdrawn when a real system menu dropdown opens at NSPopUpMenuWindowLevel (101); Liquid Glass changed menu-bar rendering and broke menu-bar managers, so layering must be re-validated on 26.5. Normal-level windows are also pushed out from under the menu bar by AppKit's frame constraining. MITIGATION: place the panel and its own dropdowns at level >=101 (NSPopUpMenuWindowLevel) or CGShieldingWindowLevel(); verify on build 25F80 that this composites above the transparent Liquid Glass menu bar; use the lowest level that achieves dominance to avoid covering system alerts/Control Center; abandon 'shift the real menu bar aside' as infeasible (no public/known-private API).
- AX NAME-STRING CONSTANTS NOT RE-EXPORTED BY THE BINDING (severity: low-medium). objc2-application-services binds the AX functions but the kAX* attribute/role/notification NAMES are CFSTR macros not exposed as symbols; a typo silently returns kAXErrorAttributeUnsupported / no notifications. MITIGATION: centralize a single verified CFString constants module (values confirmed against the local CLT SDK headers and accessibility-sys), and unit-test that AXUIElementCopyAttributeNames on a known window/menu contains them.
- NON-NATIVE / CATALYST APPS EXPOSE INCOMPLETE AX MENU TREES (severity: medium). Some cross-platform toolkits present partial or non-standard AXMenuBar trees. MITIGATION: feature-detect per app (bail cleanly if no usable AXMenuBar), render only what AX yields, and treat these under the same lazy-on-hover path as dynamic menus.

## Per-dimension findings

### Reading the focused app's menu bar tree cross-process via the Accessibility (AX) API, incl. the lazy-loading (delegate-populated submenu) problem

**Feasibility:** `feasible-with-caveats`

Enumerating a foreign app's menu bar is a well-trodden AX path and works today: AXUIElementCreateApplication(pid) -> kAXMenuBarAttribute -> AXMenuBar -> AXMenuBarItem(s) -> AXMenu -> AXMenuItem(s), recursing where an AXMenuItem itself has an AXMenu child. Per-item metadata (title, enabled, checkmark, full keyboard shortcut) is fully exposed via documented attributes. All required functions/types are available to a pure-objc2 project through objc2-application-services (features AXUIElement + HIServices). The load-bearing caveat is lazy loading: STATIC menus are readable without opening them (AppleScript's `entire contents`, System Events, VoiceOver all read them cold), but menus/submenus built on demand via NSMenuDelegate (menuNeedsUpdate:, numberOfItemsInMenu:, menu:updateItem:atIndex:shouldCancel:) — the classic being File > Open Recent, Window, Font/plug-in/Scripts menus — DO NOT EXIST in the AX hierarchy until the menu is physically opened. The only cross-process way to force population is AXPress/AXShowMenu on the parent, which visibly drops the menu in the target app and then must be dismissed; there is no documented invisible force-populate. Requires Accessibility permission; cross-process AX is synchronous IPC so large trees are slow and need batching/timeouts.

**Key APIs**
- `AXUIElementCreateApplication(pid)` (ApplicationServices/HIServices) — Create the root AXUIElement for the target app given its process id (get pid from the focused app's NSRunningApplication).
- `AXUIElementCopyAttributeValue(elem, "AXMenuBar", &out)` (HIServices) — kAXMenuBarAttribute: fetch the app's AXMenuBar element (role AXMenuBar).
- `kAXChildrenAttribute ("AXChildren")` (HIServices) — Enumerate children: AXMenuBar -> [AXMenuBarItem], AXMenuBarItem -> [AXMenu], AXMenu -> [AXMenuItem]; a submenu is an AXMenuItem whose own AXChildren contains an AXMenu.
- `kAXRoleAttribute ("AXRole")` (HIServices) — Role discrimination via constants AXMenuBar / AXMenuBarItem / AXMenu / AXMenuItem (kAXMenuBarRole, kAXMenuBarItemRole, kAXMenuRole, kAXMenuItemRole).
- `kAXTitleAttribute ("AXTitle") + kAXEnabledAttribute ("AXEnabled")` (HIServices) — Item label and enabled/greyed state; empty title typically indicates a separator.
- `kAXMenuItemMarkCharAttribute ("AXMenuItemMarkChar")` (HIServices) — The mark/checkmark character shown next to state items (e.g. checkmark).
- `kAXMenuItemCmdCharAttribute / CmdVirtualKey / CmdGlyph / CmdModifiers` (HIServices) — Keyboard shortcut: CmdChar is the literal char (e.g. C); if empty use CmdVirtualKey (keycode) or CmdGlyph (integer glyph id for non-char keys like arrows/return/delete). CmdModifiers is a bitmask.
- `AXMenuItemModifiers bitmask (Shift=1, Option=2, Control=4, NoCommand=8)` (HIServices) — Decode modifiers. NOTE the inversion vs NSEvent: Command is IMPLIED (a value of 0 means Cmd-only); bit 3 (NoCommand, value 8) set means the shortcut has NO Command key.
- `AXUIElementPerformAction(elem, "AXPress") / kAXShowMenuAction ("AXShowMenu")` (HIServices) — Trigger a mirrored item's real action; also the ONLY cross-process lever to force a lazily-built submenu to populate (opens it visibly).
- `AXUIElementCopyMultipleAttributeValues / AXUIElementCopyAttributeValues` (HIServices) — Batch-read several attributes per element in one IPC round trip — essential for walking large menu trees.
- `AXUIElementSetMessagingTimeout` (HIServices) — Bound per-message latency so an unresponsive target app can't hang Lintel's tree walk.
- `AXObserver + AXObserverAddNotification` (HIServices) — Subscribe to menu/value change notifications to re-read incrementally instead of full re-walks.
- `AXIsProcessTrustedWithOptions(kAXTrustedCheckOptionPrompt)` (HIServices) — Check/prompt for the Accessibility permission required for all cross-process AX reads (else calls return kAXErrorAPIDisabled).

**Rust mapping:** Primary: objc2-application-services (v0.3.2+) with cargo features `AXUIElement` and `HIServices`. Verified via docs.rs that it exports the AXUIElement type plus every needed function: AXUIElementCreateApplication, AXUIElementCopyAttributeValue, AXUIElementCopyMultipleAttributeValues, AXUIElementCopyAttributeValues, AXUIElementPerformAction, AXUIElementSetMessagingTimeout, AXUIElementGetPid, AXObserverCreate/AddNotification/GetRunLoopSource, AXValueCreate/GetValue, and the AXMenuItemModifiers / AXError / AXValueType enums and kAXTrustedCheckOptionPrompt. This keeps Lintel on one binding stack (objc2 + objc2-core-foundation for CFString/CFArray/CFType glue). CAVEAT: objc2-application-services does NOT re-export the AX attribute/role NAME string constants (kAXMenuBarAttribute, kAXTitleAttribute, kAXMenuItemCmdCharAttribute, kAXMenuBarItemRole, etc.) — those are plain CFStrings you must define yourself as literals ("AXMenuBar","AXTitle","AXChildren","AXMenuItemCmdChar","AXMenuItemCmdModifiers","AXMenuItemMarkChar","AXMenuBarItem","AXMenu","AXMenuItem",...). Every exact string value is confirmed in the accessibility-sys crate (eiz/accessibility) which you can copy or depend on. Alternative binding: the accessibility + accessibility-sys crates define all these string/modifier constants but are built on core-foundation-rs (not objc2/objc2-core-foundation), which would mix two CF stacks — avoid; prefer objc2-application-services + hand-declared CFString constants.

**Confirmed behaviors**
- Menu path AXMenuBar -> AXMenuBarItem -> AXMenu -> AXMenuItem with submenus as AXMenuItem-with-AXMenu-child is the standard, documented AX shape (role constants confirmed in accessibility-sys).
- Static menu trees ARE fully readable cross-process WITHOUT opening the menu (AppleScript `entire contents` / System Events enumerate them cold, including disabled items).
- Dynamically/delegate-populated menus (Open Recent, etc.) are ABSENT from the AX hierarchy until the parent menu is actually opened — macscripter: 'the elements simply don't exist in the accessibility hierarchy until the parent menu is clicked open'; workaround is click-then-poll-until-exists with delays, descending level by level.
- Shortcut modifier bitmask semantics confirmed: Shift=1, Option=2, Control=4, NoCommand=1<<3; Command is implied unless the NoCommand bit is set (inverted vs NSEvent flags).
- objc2-application-services exposes AXUIElement and the full AXUIElement*/AXObserver*/AXValue* function set including the batch (CopyMultipleAttributeValues) and timeout (SetMessagingTimeout) calls.
- All AX reads require the Accessibility permission (AXIsProcessTrustedWithOptions); without it attribute reads fail with kAXErrorAPIDisabled.

**Risks**
- (high) Lazy submenus (Open Recent, Window, Font, plug-in/Scripts menus) return nothing until opened; the mirrored bar would show them empty or stale. — *mitigation:* Detect the case (an AXMenuItem that reports a submenu-capable role/child but whose AXMenu has zero children); render it as a live passthrough that only populates on hover/click, at which point AXPress/AXShowMenu the real submenu, poll until AXChildren appears, read, then immediately dismiss (Escape / re-press). Accept a brief visible flash, or open it off-screen if the app allows. Do NOT assume full pre-read.
- (medium) Forcing population via AXPress/AXShowMenu is visibly disruptive: it drops the real menu in the target app and requires an explicit dismiss; menu tracking can also transiently affect the target. — *mitigation:* Only force-open on user intent (hovering the specific mirrored submenu), never during background pre-walks; always pair with a dismiss; consider mirroring only the top-level + first-level items eagerly and deferring deeper delegate menus.
- (medium) Walking large trees is many synchronous cross-process IPC round trips; slow apps or huge menus (Font/Recent) can stall the UI. — *mitigation:* Use AXUIElementCopyMultipleAttributeValues to batch attributes per element, set AXUIElementSetMessagingTimeout, walk on a background thread, cache per-app and refresh via AXObserver notifications instead of full re-walks.
- (medium) Non-AppKit apps (Electron/Chromium, Catalyst, some cross-platform toolkits) may expose incomplete or non-standard menu AX trees. — *mitigation:* Feature-detect per app; fall back to showing only what AX yields, or skip mirroring for apps that expose no usable AXMenuBar.
- (low) Separator detection is not a first-class AX role (separators surface as AXMenuItem with empty AXTitle), so heuristic classification can misfire. — *mitigation:* Treat empty-title, no-action, disabled AXMenuItems as separators; validate against Accessibility Inspector across sample apps and tune.
- (low) objc2-application-services lacks the kAX* attribute/role NAME string constants, so typos in hand-declared CFStrings silently return kAXErrorAttributeUnsupported. — *mitigation:* Copy the exact strings from accessibility-sys and centralize them in one constants module with tests against Accessibility Inspector output.

**Critical claim:** The riskiest load-bearing assumption is that the target app's menu tree can be read in full via AX before the user opens each menu. This holds for statically-defined menus but FAILS for NSMenuDelegate/lazily-built submenus (Open Recent, Window, Font, plug-in menus), whose AXMenuItem children do not exist in the AX hierarchy until the menu is physically opened — and the only cross-process way to force population (AXPress/AXShowMenu) visibly opens the real menu in the target app, which conflicts with Lintel's goal of an unobtrusive mirrored bar.

**Adversarial verdicts:**
- `confirmed-with-caveats` (confidence medium) — The claim has a legitimate, load-bearing kernel but mis-scopes it and picks wrong examples, so it is not accurate as stated.

WHAT IS CORRECT: (1) Static NIB/storyboard menus are fully AX-readable before opening. (2) There genuinely exists a class of menus whose AXMenuItem children do NOT appear in the AX tree until display. Apple's documented semantics for the delegate hook are exactly this: NSMenuDelegate menuNeedsUpdate: is "Invoked when a menu is about to be displayed at the start of a tracking session" (Apple docs, quoted in the JUCE thread). So menus that add their items only in menuWillOpen: / menuNeedsUpdate:, plus non-native/custom-drawn menus (Electron, Java, some plug-in hosts, Logic Pro-style custom UI) that have no backing NSMenu, will return empty AXChildren until opened. (3) The cross-process trigger for those (AXPress on the AXMenuBarItem, AXShowMenu) does visibly open the real menu on screen, which conflicts with an unobtrusive mirror. Real-world experts confirm the native-vs-nonnative split: for traditional AppKit menus "all menu items are kept in active memory ... without having to be visible on screen," while a non-native app "requires you to click ... to have access to its menu items."

WHAT IS WRONG / MISLEADING: (a) The examples are largely incorrect. Window (NSApplication addWindowsItem:), Font (NSFontManager), and Open Recent (NSDocumentController) are AppKit-managed menus whose items are REAL, persistent NSMenuItems maintained as app state changes — not rebuilt-on-open delegate menus — so they ARE enumerable via AXChildren without opening. These are among the WORST examples of the failure mode; good examples are menuWillOpen:-built menus and non-native/plug-in custom menus. (b) The blanket "all NSMenuDelegate/lazily-built" is too strong — a menu with a delegate is not automatically empty until opened; only items built at tracking-start are. (c) "the only cross-process way to force population ... visibly opens" is overstated: AppKit also runs -[NSMenu update] on non-display paths, so some dynamic menus populate for AX without a visible open; the visible-open constraint is firm only for menuWillOpen:-built and non-native menus.

CONFIDENCE is medium because Apple's live doc pages are JS-rendered and could not be fetched verbatim (I relied on an Apple-doc quote reproduced in a developer forum plus AppKit API knowledge and expert corroboration), and I could not obtain a crisp Apple statement on whether AXChildren access triggers menuNeedsUpdate for every dynamic-menu variant.
  - corrected: A genuinely load-bearing risk is that not every menu tree is AX-readable before opening — but the failure is narrower and differently-scoped than stated. Static NIB menus AND dynamically-maintained AppKit menus backed by persistent real NSMenuItems (Window via NSApplication addWindowsItem:, Font via NSFontManager, Open Recent via NSDocumentController) ARE enumerable via AXChildren without opening, so those are poor examples. The true failure cases are: (1) menus whose items are built only at display time in NSMenuDelegate.menuWillOpen: / menuNeedsUpdate: (Apple: "invoked when a menu is about to be displayed at the start of a tracking session"), and (2) non-native or custom-drawn menus (Electron, Java, Qt, some audio plug-in hosts) that have no backing NSMenu. For those, AXChildren returns empty until a tracking session begins, and the cross-process way to force it — AXPress on the AXMenuBarItem or performing AXShowMenu — visibly opens the real menu on screen, conflicting with an unobtrusive mirror. Caveat: this is not universally the ONLY path, since AppKit runs -[NSMenu update] on some non-display code paths; the visible-open constraint is firm specifically for menuWillOpen:-populated and non-native menus. Lintel should therefore pre-read statically what it can and lazily mirror problematic submenus on hover by mapping to the target's AXMenuItem (or replay the key equivalent) rather than assuming the whole tree is pre-readable.
- `confirmed-with-caveats` (confidence medium) — The phenomenon is real but the claim is overstated and misidentifies which menus fail. TRUE: genuinely lazy menus that build contents only when opened (populated in NSMenuDelegate menuWillOpen:, non-AppKit toolkits like Electron/Qt/JUCE, or custom popup controls) do not expose child items in the cross-process AX tree until opened, and the documented actions to open them (kAXPressAction/kAXShowMenuAction) visibly display the real menu. Authoritative AppleScript UI-scripting expert Chris Stone (CJK) confirms: for such menus 'whatever is not displayed on screen physically does not exist at that point in time' and 'you need to click the menu in order to have access to its menu items,' with UI Browser failing to capture the path (keyboardmaestro/29039). MISLEADING: the flagship examples are mostly NOT failure cases. The same expert says NATIVE macOS menus keep 'all menu items in active memory to be triggered directly without having to be visible on screen.' The Window menu is read directly via `menu 1 of menu bar item Window` without opening (macscripter/66233); macOS Help/Spotlight search reads deep native submenus like Format>Font>Baseline>Superscript without opening (macmost); the Format>Font submenu is a static AppKit menu (the dynamic font list is in the separate Font Panel); Open Recent is AppKit-managed and generally readable (an older thread found it flaky and suggested com.apple.recentitems.plist, macscripter/37655). So Window and Font are wrong examples, Open Recent borderline, only genuine plug-in menus reliably fail. The claim also conflates all NSMenuDelegate menus: AppKit's -[NSMenu update] runs the menuNeedsUpdate:/numberOfItemsInMenu: lazy pattern and native delegate menus using it are generally AX-readable without a visible open; only menuWillOpen:-built or non-native menus truly require opening. I found no definitive Apple citation that a cross-process AX children query itself triggers menuNeedsUpdate: without display, hence medium confidence. Mitigations exist (mirror eagerly-exposed native menus; for rare lazy submenus trigger AXShowMenu on hover of the mirrored parent, or read backing stores like the recent-items plist), so it is not clearly the single riskiest assumption.
  - corrected: Reading the full menu tree via AX before opening each menu works for the vast majority of menu-bar menus in native AppKit apps (File/Edit/View/Format and their submenus, the Window menu, the Format>Font submenu, and usually Open Recent), because AppKit keeps items in memory / updates them via NSMenu -update (menuNeedsUpdate:) and exposes them to the Accessibility API without visibly displaying the menu. It genuinely FAILS only for a narrower set: menus built on demand in NSMenuDelegate menuWillOpen: (not menuNeedsUpdate:), some third-party plug-in menus, and menus in non-AppKit toolkits (Electron, Qt, Java/JUCE) or custom popup controls; for these the AXMenuItem children are absent from the cross-process AX tree until opened, and kAXPressAction/kAXShowMenuAction visibly open the real menu. A real but bounded risk, and it does not apply to the Window/Font examples cited.

**Open questions**
- Does a passive AXChildren read on an AXMenu ever trigger -[NSMenu update]/menuNeedsUpdate: for some AppKit apps (partial population), or is opening strictly required for all delegate-driven menus? Needs empirical testing with Accessibility Inspector across apps.
- When AXPress-ing a submenu to force population, can it be opened/dismissed fast enough to avoid a perceptible flash, and does opening the real menu ever steal focus or interfere with the target (relevant to the non-activating requirement)?
- Are cross-process AXUIElement calls safe/robust from a background thread in macOS 26, or should the tree walk run on the main thread with a run loop? Sources conflict; verify on target OS.
- How reliably do AXObserver menu-change notifications fire for menu content changes (vs only open/close), to keep the mirror fresh without full re-walks?
- Exact separator representation and how to distinguish a true separator from a blank/placeholder AXMenuItem across toolkits.

### Triggering a target app's menu item action from our separate process (AXUIElementPerformAction / kAXPressAction on AXMenuItem vs. synthesizing key-equivalents via CGEvent)

**Feasibility:** `feasible-with-caveats`

Firing another app's menu command from Lintel is feasible and the accessibility (AX) route is clearly the right primary mechanism. AXUIElementPerformAction(element, kAXPressAction) on a *leaf* AXMenuItem invokes that item's action via an IPC messaging transaction to the target app — it does NOT go through the event/focus system, so it does NOT require the target to be frontmost and does NOT steal focus (this directly satisfies Lintel's non-activating requirement). Pressing a leaf item generally does NOT visually open/animate the menu bar (no flash). The important caveats are all about *finding* the AXMenuItem reference and about lazily-built menus: (1) AX-pressing a top-level AXMenuBarItem or a submenu parent DOES visually open that menu, so you must target the leaf item directly, not walk it open; (2) many apps populate submenu contents lazily (NSMenuDelegate/menuNeedsUpdate:), so the AXMenu's AXChildren can be empty/stale until the menu has been opened or validated at least once, and some apps only set an item's enabled state when the menu opens — for those you may have to briefly open (AXShowMenu / AXPress the parent) to populate, which can cause a flash; (3) AXUIElementPerformAction is a synchronous cross-process call with a ~6s default timeout, so it must be run off the UI thread and/or with a shortened messaging timeout. The CGEvent key-equivalent alternative is markedly worse for this use case: CGEventPostToPid/PSN delivery to a non-focused app is app-dependent and frequently ignored unless the app is key/active (which would steal focus and mutate the very menu we mirror), CGEventPostToPSN is deprecated, and most menu items have no key-equivalent at all, so it cannot cover the full menu. CGEvent should be a narrow fallback, not the primary path.

**Key APIs**
- `AXUIElementPerformAction(element, action)` (ApplicationServices / HIServices (Accessibility)) — Primary trigger: pass kAXPressAction on a leaf AXMenuItem to fire its action. Synchronous IPC, returns AXError; can time out (default 6s).
- `kAXPressAction` (Accessibility constants) — The default 'press' action string; on a menu item = perform its command, on a menu-bar title/submenu parent = open that menu (visible).
- `AXUIElementCopyAttributeValue + kAXChildrenAttribute` (Accessibility) — Traverse AXMenuBar -> AXMenuBarItem -> AXMenu -> AXMenuItem to locate the target item. Children of lazily-built menus may be empty until the menu is validated/opened.
- `kAXMenuBarRole / kAXMenuBarItemRole / kAXMenuRole / kAXMenuItemRole` (Accessibility role constants) — Identify nodes while walking the menu hierarchy; distinguish top-level titles from leaf items.
- `kAXMenuItemCmdCharAttribute / kAXMenuItemCmdModifiersAttribute / kAXMenuItemCmdVirtualKeyAttribute` (Accessibility) — Read an item's key-equivalent (if any) — needed only for the CGEvent fallback and to know whether one even exists.
- `AXShowMenu action` (Accessibility) — Explicitly open a menu (needed for Dock/contextual menus and to force-populate lazy submenus); causes visible display — avoid unless required.
- `AXUIElementSetMessagingTimeout` (Accessibility) — Lower the default 6s per-element timeout so a hung target app doesn't block Lintel.
- `AXUIElementCreateApplication(pid) / AXIsProcessTrustedWithOptions` (Accessibility) — Get the target app's AX root by pid; check/prompt for the required Accessibility (AXTrust) permission.
- `CGEventCreateKeyboardEvent + CGEventPostToPid (and legacy CGEventPostToPSN)` (CoreGraphics) — Fallback only: synthesize the item's key-equivalent to a target process. Unreliable for non-focused apps; PSN variant deprecated.

**Rust mapping:** Use the accessibility-sys crate (v0.2.x, MIT/Apache-2.0, from eiz/accessibility). Its docs.rs listing confirms it exposes AXUIElementPerformAction, AXUIElementCopyAttributeValue, AXUIElementCreateApplication, the AXObserver/AXValue families, AXIsProcessTrusted*, and the full kAX* constant set (kAXPressAction, kAXMenuItemRole, kAXMenuBarRole, kAXChildrenAttribute, etc.). Constant string values and function signatures live on the per-item detail pages (crate is 0% documented but complete); AXUIElementPerformAction is (element: AXUIElementRef, action: CFStringRef) -> AXError. It depends on core-foundation-sys 0.8, which composes with objc2-core-foundation used elsewhere in Lintel (be careful mixing CFTypeRef types across core-foundation-sys vs objc2-core-foundation — wrap/transmute at the boundary). The companion high-level `accessibility` crate exists but is described by its own author as 'spotty', so drive accessibility-sys directly for menu work. For the CGEvent fallback, use core-graphics/core-graphics-sys or objc2-core-graphics for CGEventCreateKeyboardEvent/CGEventPost; verify CGEventPostToPid is actually bound (it is present in core-graphics-sys; CGEventPostToPSN may not be exposed and is deprecated). Accessibility permission (AXIsProcessTrustedWithOptions with kAXTrustedCheckOptionPrompt) is mandatory for any of this to work.

**Confirmed behaviors**
- AXUIElementPerformAction issues a synchronous cross-process accessibility messaging transaction with a default ~6s timeout, and may return before the action's UI effects complete (Apple docs).
- AX actions invoke an element's default action programmatically and do NOT inherently require the target app to be frontmost/active — a core advantage over synthetic input (community + API semantics).
- Dock/contextual menus require an explicit AXShowMenu before their items can be enumerated/pressed (Keyboard Maestro forum: 'You've got to get the menu open before you can select items in it').
- accessibility-sys exposes AXUIElementPerformAction and the kAXPressAction/kAXMenuItemRole/kAXMenuBarRole/kAXChildrenAttribute constants (docs.rs listing).
- CGEventPostToPid delivery to a non-focused/background app is app-dependent and often ignored unless the app is brought to focus; events crafted to match an app's registered menu key-equivalent are more reliable than generic keystrokes but still not universal (Apple Developer Forums threads).
- CGEventPostToPSN is the historical way to target a specific process but is deprecated.

**Risks**
- (medium) Lazily-populated menus: apps that build submenu contents via NSMenuDelegate/menuNeedsUpdate: may expose empty AXChildren until the menu is opened once, so the leaf AXMenuItem to press may not exist yet. — *mitigation:* Mirror menus opportunistically; if children are empty, issue AXShowMenu / AXPress the parent to force population (accepting a brief flash) or observe kAXMenuOpenedNotification to cache structure. Cache the populated hierarchy per app.
- (medium) Item enabled-state / validation is only computed when the real menu opens for some apps, so an AXPress may be a no-op or the item may report disabled until opened once. — *mitigation:* Read kAXEnabledAttribute before pressing; if unreliable, do a one-time hidden open to prime validation, or fall back to key-equivalent for items known to have one.
- (low) Visible flash if you accidentally AXPress a top-level AXMenuBarItem or a submenu parent (their default action is to OPEN the menu). — *mitigation:* Always resolve to and AXPress the LEAF AXMenuItem directly; never press ancestors as a navigation step.
- (low) Synchronous 6s timeout can block Lintel's UI thread if the target app is busy/unresponsive. — *mitigation:* Call AXUIElementPerformAction off the main thread and set a short AXUIElementSetMessagingTimeout (e.g. 1-2s).
- (medium) CGEvent fallback steals focus / can't cover items without key-equivalents, and reliability varies by app. — *mitigation:* Keep CGEvent strictly as a last-resort per-item fallback only for items that have a key-equivalent AND fail AXPress; prefer AX everywhere else.
- (high) Requires Accessibility (AXTrust) permission; without it every AX and CGEvent call fails silently/erroring. — *mitigation:* Gate the feature behind AXIsProcessTrustedWithOptions prompt and clear onboarding; feature is non-functional without it (inherent OS requirement, not a bug).
- (low) core-foundation-sys (accessibility-sys) vs objc2-core-foundation type mismatch at the FFI boundary. — *mitigation:* Isolate AX FFI in one module and convert CFTypeRef pointers explicitly at the boundary rather than passing objc2 types directly.

So AX is the right primary path, but 'cleanly, every app, first try, zero flash' is not universally guaranteed.

### Detecting the focused app and focused window, its on-screen frame, and reacting to changes (app/window focus, geometry, fullscreen/zoomed state).

**Feasibility:** `feasible-with-caveats`

The full detection pipeline is feasible on macOS 26.5 with objc2, using entirely documented APIs except one private attribute. Flow: (1) AppKit NSWorkspace notifications tell you which application is frontmost; (2) per-app AXObserver + Accessibility (AX) notifications tell you which window is focused and when it moves/resizes/minimizes/is destroyed; (3) AXUIElementCopyAttributeValue with kAXPosition/kAXSize gives the window frame in the AX (top-left origin, y-down) coordinate space, which you convert to Cocoa/NSScreen coordinates by flipping against the primary display height. Native-fullscreen detection works via the UNDOCUMENTED "AXFullScreen" attribute (the only non-public piece). Zoomed/maximized-but-not-fullscreen is inferred by comparing the (converted) window frame to NSScreen.visibleFrame. All AX access requires the app to be granted Accessibility (AXIsProcessTrusted) permission. The Rust bindings all exist in objc2-application-services + objc2-core-foundation + objc2-app-kit, but the AX attribute/notification name constants are NOT exposed by the crate (they are CFSTR macros in the SDK), so you must build the CFStrings from string literals yourself.

**Key APIs**
- `NSWorkspace.shared.notificationCenter` (AppKit (objc2-app-kit)) — The DEDICATED notification center that posts workspace app-lifecycle events. NOTE: not NSNotificationCenter.default — registering on the default center receives nothing.
- `NSWorkspaceDidActivateApplicationNotification` (AppKit (objc2-app-kit)) — Posted when an app becomes frontmost. userInfo[NSWorkspaceApplicationKey] is the NSRunningApplication. Also NSWorkspaceDidDeactivateApplicationNotification and NSWorkspaceActiveSpaceDidChangeNotification (space switch, no app payload — useful to re-evaluate/hide the bar).
- `NSRunningApplication.processIdentifier` (AppKit (objc2-app-kit)) — pid_t of the activated app; the seed for AXUIElementCreateApplication and AXObserverCreate.
- `AXIsProcessTrusted / AXIsProcessTrustedWithOptions(kAXTrustedCheckOptionPrompt)` (ApplicationServices/HIServices (objc2-application-services)) — Gate: AX APIs return kAXErrorAPIDisabled/cannot-observe until the user grants Accessibility permission. WithOptions can prompt the user. kAXTrustedCheckOptionPrompt IS exposed as a static in the crate.
- `AXUIElementCreateApplication(pid_t)` (objc2-application-services) — Application-level AXUIElement — the root to observe and to read kAXFocusedWindow/kAXMainWindow/kAXWindows.
- `AXUIElementCreateSystemWide()` (objc2-application-services) — System-wide element; read kAXFocusedApplicationAttribute / kAXFocusedUIElementAttribute to get the current focus without per-app polling.
- `AXUIElementCopyAttributeValue(element, cfstring, &out)` (objc2-application-services) — Read attributes: kAXFocusedWindow('AXFocusedWindow'), kAXMainWindow('AXMainWindow'), kAXPosition('AXPosition'), kAXSize('AXSize'), kAXFullScreen('AXFullScreen', PRIVATE), kAXMinimized('AXMinimized'), kAXTitle, kAXWindows.
- `AXValueGetValue(axvalue, kAXValueCGPointType|kAXValueCGSizeType, &out) / AXValueGetType` (objc2-application-services) — Unwrap the AXValue boxes returned for kAXPosition (CGPoint) and kAXSize (CGSize). kAXValueCGPointType/CGSizeType/CGRectType/CFRangeType ARE exposed as statics in the crate.
- `AXObserverCreate(pid, callback, &observer) / AXObserverCreateWithInfoCallback` (objc2-application-services) — Create a per-app observer. Rust sig: unsafe extern "C-unwind" fn(pid_t, AXObserverCallback, NonNull<*mut AXObserver>) -> AXError. Idiomatic wrapper AXObserver::create also exists. refcon carries your Rust state pointer.
- `AXObserverAddNotification(observer, element, cfstring, refcon) / AXObserverRemoveNotification` (objc2-application-services) — Subscribe the observer to named AX notifications on an element (app element for activation/focus-change; specific window element for move/resize/destroy).
- `AXObserverGetRunLoopSource(observer) + CFRunLoopAddSource(CFRunLoopGetCurrent(), src, kCFRunLoopDefaultMode)` (objc2-application-services + objc2-core-foundation) — MANDATORY: no callbacks fire until the observer's CFRunLoopSource is added to a running run loop (use main thread / main run loop). Remove on teardown.
- `AXUIElementSetMessagingTimeout(element, seconds)` (objc2-application-services) — Guard against slow/hung target apps blocking your AX reads (Electron/Java apps can be slow).
- `NSScreen.screens / .frame / .visibleFrame` (AppKit (objc2-app-kit)) — Primary-display height for AX->Cocoa y-flip; visibleFrame (excludes menu bar/Dock) to detect zoomed/maximized-but-not-fullscreen by frame comparison.

**Rust mapping:** objc2-application-services (all AX* functions/types: AXUIElement, AXObserver, AXValue, AXError, plus fns confirmed present on docs.rs — AXObserverCreate, AXObserverAddNotification/RemoveNotification, AXObserverGetRunLoopSource, AXUIElementCreateApplication, AXUIElementCreateSystemWide, AXUIElementCopyAttributeValue, AXUIElementCopyAttributeNames, AXValueGetValue/GetType, AXUIElementSetMessagingTimeout, AXIsProcessTrusted/WithOptions, AXUIElementPerformAction). Fine-grained cargo features gate each item, e.g. AXObserverCreate needs features = ["AXError","AXUIElement","HIServices","libc"]; enable AXValue for AXValueGetValue, plus the umbrella objc2 feature. objc2-core-foundation supplies CFString (build the AX name strings from literals), CFRunLoop/CFRunLoopSource, CFRunLoopGetCurrent, CFRunLoopAddSource/RemoveSource, kCFRunLoopDefaultMode/kCFRunLoopCommonModes, CGPoint/CGSize/CGRect. objc2-app-kit supplies NSWorkspace, NSRunningApplication, NSScreen. objc2-foundation supplies NSString/NSNotification. CRITICAL BINDING GAP: the crate does NOT export the AX attribute/notification name constants (kAXPositionAttribute, kAXFocusedWindowChangedNotification, kAXFullScreenAttribute, etc.) because in the C SDK they are `#define kFoo CFSTR("...")` macros, not extern symbols that bindgen can translate — only the text-attribute externs, AXValueType enum, and kAXTrustedCheckOptionPrompt are exposed. You must construct CFString/NSString from the literal string values yourself.

**Confirmed behaviors**
- NSWorkspace app-activation notifications are posted on NSWorkspace.shared.notificationCenter (a separate center), with the NSRunningApplication under userInfo key NSWorkspaceApplicationKey (Apple docs).
- AXObserverAddNotification takes a CFString notification name and a refcon; AXObserverCallback signature is (AXObserver, AXUIElement, CFString, refcon)->void; the WithInfo variant adds a CFDictionary (Apple ApplicationServices docs).
- Notifications only arrive after AXObserverGetRunLoopSource's source is added to a run loop via CFRunLoopAddSource (Apple docs).
- The AX/Quartz global coordinate space uses a top-left origin with y increasing downward, opposite to Cocoa's bottom-left origin; convert with y_cocoa = primaryScreenHeight - y_ax (flip against primary display height), x unchanged (SDL issue #1576, confirmed against known AppKit semantics).
- Window position/size are read as boxed AXValue via kAXPositionAttribute('AXPosition')/kAXSizeAttribute('AXSize') and unwrapped with AXValueGetValue using kAXValueCGPointType/kAXValueCGSizeType (yabai src/window.c window_ax_origin/window_ax_frame).
- Native fullscreen is detectable via the attribute string 'AXFullScreen' — yabai defines `const CFStringRef kAXFullscreenAttribute = CFSTR("AXFullScreen");` in src/window.h and reads it with AXUIElementCopyAttributeValue in window_is_fullscreen.
- Standard AX notification constants used by real window managers: kAXUIElementDestroyedNotification, kAXWindowMiniaturizedNotification, kAXWindowDeminiaturizedNotification, kAXWindowMoved, kAXWindowResized, kAXFocusedWindowChanged, kAXMainWindowChanged, kAXApplicationActivated, kAXFocusedUIElementChanged — string values 'AXUIElementDestroyed','AXWindowMiniaturized','AXWindowMoved','AXWindowResized','AXFocusedWindowChanged','AXMainWindowChanged','AXApplicationActivated','AXFocusedUIElementChanged' (yabai src/window.h + AX-notifications gist).
- objc2-application-services on docs.rs exposes all required AX functions and types (verified on the crate's all.html and per-fn pages).

**Risks**

### Non-activating floating overlay window (NSPanel) that receives clicks without stealing focus from the target app; accessory activation policy

**Feasibility:** `feasible`

The critical claim is CONFIRMED: an NSPanel created with the NSWindowStyleMask.nonactivatingPanel style mask receives mouse (and, if made key, keyboard) events WITHOUT activating its owning application, so the previously-frontmost target app stays frontmost. Pairing this with NSApplication.setActivationPolicy(.accessory) (or Info.plist LSUIElement=1) removes the Dock icon and the app's own main menu, and guarantees the app is never treated as a regular foreground app. An .accessory app runs a completely normal NSApplication event loop and can present fully clickable UI — this is exactly how menu-bar/status-item apps work. All required APIs have direct, verified objc2-app-kit bindings. The only real gotchas are (a) a borderless/panel window that you also want to accept KEYBOARD input into (e.g. a menu search field) must override canBecomeKeyWindow to return true, and (b) to make the very FIRST click on a control actionable while the panel is inactive, content views should override acceptsFirstMouse(for:) -> true. Neither blocks clicks; mouse-down events are delivered to a non-active, non-key window regardless. Note: actually triggering the *target app's* real menu action from a mirrored click is a separate dimension (Accessibility/AXUIElement or menu re-dispatch) — the non-activating panel is fully compatible with it, but that mechanism is out of scope here.

**Key APIs**
- `NSWindowStyleMask::NonactivatingPanel (value 128)` (objc2-app-kit) — Core mechanism: a panel with this style mask does NOT activate the owning app when clicked / when it becomes key. This is what keeps the target app frontmost.
- `NSPanel (objc2_app_kit::NSPanel)` (objc2-app-kit) — Window subclass used for the overlay; combine with NonactivatingPanel (+ usually Borderless) style mask.
- `NSPanel::setFloatingPanel / isFloatingPanel` (objc2-app-kit) — Marks the panel as a floating utility panel (auto-orders above normal windows, hides on app deactivate unless overridden).

- `NSApplication::setActivationPolicy(NSApplicationActivationPolicy) -> bool` (objc2-app-kit (crate feature NSRunningApplication)) — Call with NSApplicationActivationPolicy::Accessory early in main() so the app has no Dock icon / no menu and is never the regular frontmost app.

- `NSView::acceptsFirstMouse(for:) -> bool (override)` (objc2-app-kit) — Return true so the first click on a mirrored control acts immediately instead of just ordering the inactive panel forward.

### Can an ordinary app window be drawn ON TOP OF the system menu bar on the primary display (covering it), and can the menu bar be shifted aside?

**Feasibility:** `feasible-with-caveats`

COVERING the menu bar is feasible with public API. The system menu bar is drawn at NSWindow level mainMenu (integer 24). Any ordinary window whose level is set higher than 24 and positioned in the menu-bar region will render ON TOP of it — the WindowServer does NOT clip ordinary app windows out of the menu-bar rectangle; the menu bar is simply another window layer in the z-order. Practical levels above it: statusBar (25), popUpMenu (101), screenSaver (NSWindow constant = 101), or CGShieldingWindowLevel() (~2147483628, the level the system itself uses to shield the whole screen incl. menu bar). This works from a background/accessory (LSUIElement) app too, because window-level z-ordering is independent of app activation — so covering does NOT conflict with Lintel's non-activating requirement and does NOT require presentationOptions. Caveat: the real menu bar's own pull-down menus and Control Center draw at popUpMenu level (101), so if the user clicks a real system menu it can appear above a bar placed at level 25; place Lintel's bar (and its own dropdowns) high enough (>=101, or use a dedicated high level) to stay visually dominant, but avoid going so high you cover system-critical alerts. SHIFTING the system menu bar's contents horizontally is INFEASIBLE: there is no public or known-private API to reposition the Apple menu / app menus / status items horizontally. Auto-HIDING the whole system menu bar via NSApplication.presentationOptions (.autoHideMenuBar/.hideMenuBar) is possible in principle but presentationOptions are honored only for the ACTIVE (frontmost) app and .hideMenuBar must be combined with .hideDock — which both fight Lintel's non-activating design. Recommended path: just cover the menu-bar region with a high-level window; do not try to hide or move the real bar.

**Key APIs**
- `NSWindow.level / setLevel:` (AppKit (objc2-app-kit)) — Set the window's z-order level above mainMenu (24) so it renders over the system menu bar.
- `NSMainMenuWindowLevel (=24), NSStatusWindowLevel (=25), NSPopUpMenuWindowLevel (=101), NSScreenSaverWindowLevel (=101)` (AppKit) — The menu bar sits at mainMenu(24); levels above it cover it. statusBar(25) covers the static bar; popUpMenu(101) also sits above real dropdown menus.
- `CGShieldingWindowLevel()` (CoreGraphics (objc2-core-graphics)) — Returns the window level the system uses to shield the entire display including the menu bar — the definitive 'cover everything' level.
- `CGWindowLevelForKey(kCGMainMenuWindowLevelKey / kCGPopUpMenuWindowLevelKey / kCGScreenSaverWindowLevelKey)` (CoreGraphics) — Programmatic source of truth for menu-bar and above-menu-bar level integers (note kCGScreenSaverWindowLevelKey=1000, diverging from NSWindow's screenSaver=101).
- `NSScreen.frame vs NSScreen.visibleFrame` (AppKit) — frame includes the menu-bar strip; visibleFrame excludes it — use frame (top area) to position the covering bar over the menu-bar rectangle.
- `NSScreen.safeAreaInsets.top / auxiliaryTopLeftArea / auxiliaryTopRightArea` (AppKit (macOS 12+)) — Notch geometry on built-in displays: menu-bar height and the usable regions left/right of the notch, for correctly sizing/splitting a bar drawn over the top.
- `NSApplication.presentationOptions / NSApplicationPresentationOptions (.autoHideMenuBar, .hideMenuBar, .hideDock)` (AppKit) — Can hide the real menu bar, but only while the app is active and .hideMenuBar requires .hideDock — flagged as conflicting with a non-activating accessory design.
- `NSApplication.setActivationPolicy(.accessory) / LSUIElement` (AppKit) — Run as a background/agent app with no Dock icon; covering the menu bar works fine in this mode since level z-order is activation-independent.

**Rust mapping:** objc2-app-kit exposes NSWindow with the msg_send equivalent `window.setLevel(level)` where NSWindowLevel is a type alias for isize (NSInteger). The level constants are provided as free constants in objc2_app_kit: NSNormalWindowLevel, NSFloatingWindowLevel, NSMainMenuWindowLevel, NSStatusWindowLevel, NSModalPanelWindowLevel, NSPopUpMenuWindowLevel, NSScreenSaverWindowLevel (each an NSWindowLevel/isize). Positioning uses NSWindow::setFrame_display: / initWithContentRect over NSScreen::frame (objc2_app_kit NSScreen::frame() -> NSRect; also visibleFrame(), safeAreaInsets() -> NSEdgeInsets, auxiliaryTopLeftArea()/auxiliaryTopRightArea() -> Option<NSRect> on macOS 12+). CGShieldingWindowLevel() and CGWindowLevelForKey() live in objc2-core-graphics (CoreGraphics) as free fns returning CGWindowLevel (i32); cast to NSWindowLevel (isize) when calling setLevel. presentationOptions maps to NSApplication::presentationOptions()/setPresentationOptions() with the NSApplicationPresentationOptions bitflags type in objc2_app_kit. Activation policy via NSApplication::setActivationPolicy(NSApplicationActivationPolicy::Accessory). No private APIs required for the covering path.

**Confirmed behaviors**
- The system menu bar is rendered at NSWindow level mainMenu = integer 24; levels statusBar=25, popUpMenu=101, screenSaver(NSWindow)=101 are numerically higher and always draw in front.
- NSWindow z-order is strictly by level then order-within-level; a higher-level window covers the menu bar rather than being clipped — the WindowServer does not carve app windows out of the menu-bar rectangle.
- CGShieldingWindowLevel() exists specifically to return a level above the menu bar to shield the whole screen (used by screen capture/login), proving the menu bar is coverable by a normal window.
- Multiple shipping apps cover or replace the top strip: 'barik' (agent app that draws its own bar and instructs the user to hide the system menu bar), 'screen-cover' (covers an entire monitor incl. menu bar), DynamicNotchKit/notch utilities (draw a custom window over the top/notch region), and menu-bar-mirroring apps like NextMenuBar.
- Covering works from an accessory/LSUIElement background app without activating it, because level z-order is independent of which app is frontmost.
- presentationOptions.hideMenuBar must be combined with hideDock and only takes effect while the app is active/frontmost; the system reverts when the app deactivates.
- macOS 26 (Tahoe/Liquid Glass) changed only menu-bar appearance (transparent vs. optional solid background); no evidence it changed window-level layering that would prevent a higher-level window from covering the bar.

**Risks**
- (medium) Real system menu pull-downs and Control Center render at popUpMenu level (101); a covering bar at only statusBar level (25) would be visually overtaken when a real menu opens. — *mitigation:* Place Lintel's bar and its own submenu windows at popUpMenu level (101) or a dedicated high level so they stay above the real bar's chrome; since Lintel mirrors actions the real menu shouldn't be opened by the user anyway.
- (medium) Using an excessively high level (CGShieldingWindowLevel / screenSaver=1000 via CG key) could cover system alerts, notifications, or the Control Center panel, harming UX or trapping the user. — *mitigation:* Use the lowest level that achieves coverage (>=101 for menu dominance). Reserve shielding-level only if strictly necessary and test alert/notification interplay.
- (low) On notched built-in displays the top strip includes the physical notch; a naive full-width bar draws content behind an invisible cutout. — *mitigation:* Use NSScreen.safeAreaInsets/auxiliaryTopLeftArea/auxiliaryTopRightArea to lay the bar out around the notch. Note the user's primary target is an external super-ultrawide (no notch), so this is edge-case.
- (high) Auto-hiding/moving the real menu bar via presentationOptions requires activation and hideDock, conflicting with the non-activating requirement, and could also be seen as user-hostile. — *mitigation:* Do NOT attempt to hide or shift the system menu bar; only visually cover its region with a higher-level window. Abandon the 'shift menu bar aside' sub-goal as infeasible.
- (low) Future macOS builds could alter menu-bar window levels or add clipping; behavior is empirical, not contractually guaranteed by Apple. — *mitigation:* Pin behavior with a runtime probe (query menu-bar level via CGWindowListCopyWindowInfo) and degrade gracefully; re-test each macOS major release.

**Critical claim:** CONFIRMED (with caveats): An ordinary app window CAN render on top of the primary display's menu-bar region. The menu bar occupies NSWindow level mainMenu (24) and is not clipped-protected; a window at any higher level (statusBar 25, popUpMenu 101, or CGShieldingWindowLevel()) positioned over NSScreen.frame's top strip draws over it, including from a non-activating accessory app. The corollary claim — that the menu bar can be shifted horizontally — is NOT confirmed and appears infeasible: no public or known-private API repositions the Apple/app menus or status items; existing 'menu bar managers' only hide/rearrange right-side status items, never the app menus, and typically rely on covering or the user manually enabling auto-hide.

### Achieving a blurred/acrylic ("frosted glass") look for the floating local-menu bar on macOS 26 (Liquid Glass), via public AppKit and the objc2-app-kit bindings.

**Feasibility:** `feasible`

A convincing acrylic/blur bar is achievable with PUBLIC AppKit on macOS 26, and the required classes are already bound in objc2-app-kit 0.3.2. Two complementary options exist: (1) the long-standing NSVisualEffectView (public since 10.10) with a blur material and blendingMode=.behindWindow, which samples/blurs whatever is behind the panel — the classic "vibrancy/acrylic" look; and (2) the NEW, public macOS 26.0+ NSGlassEffectView (plus NSGlassEffectContainerView) that renders the native "Liquid Glass" material with a first-class cornerRadius, tintColor, and style. For a bar that visually belongs on macOS 26, NSGlassEffectView is the modern, on-brand choice; NSVisualEffectView is the robust, well-understood fallback and the closest analog to Windows "Acrylic." Both are hostable inside a borderless, transparent NSPanel (isOpaque=false, clear backgroundColor). NSGlassEffectView is public API (documented on developer.apple.com and mirrored in Microsoft's macOS 26.0 binding metadata) — no private API required. Note: NSVisualEffectView materials were themselves restyled toward the glassier look under Tahoe/26, so even the fallback reads correctly on 26.

**Key APIs**
- `NSGlassEffectView` (AppKit (objc2-app-kit, feature "NSGlassEffectView")) — macOS 26.0+ PUBLIC Liquid Glass view. Bound methods: contentView/setContentView, cornerRadius/setCornerRadius (needs objc2-core-foundation for CGFloat), tintColor/setTintColor (needs NSColor feature), style/setStyle. Put the bar's menu-item content in contentView; "only guarantees the contentView will be placed inside the glass effect" so do not rely on z-order of arbitrary sibling subviews.
- `NSGlassEffectContainerView` (AppKit (objc2-app-kit, feature "NSGlassEffectView")) — macOS 26.0+ PUBLIC. Merges nearby descendant glass views into one sampled effect (contentView, spacing:CGFloat). Essential if each mirrored menu item is its own glass pill — without a container, two glass views too close "might try to sample each other" and render incorrectly (per WWDC25 lab).

### The Rust binding story (objc2 ecosystem + Accessibility C API bindings) for Lintel on macOS 26.5

**Feasibility:** `feasible`

Both halves of Lintel's binding needs are covered by the actively-maintained madsmtm/objc2 crate family as of mid-2026. AppKit is bound by objc2-app-kit 0.3.2 (NSApplication/main loop, NSPanel/NSView, styles, levels), and the full Accessibility C API is bound by objc2-application-services 0.3.2 — which was NOT true in 2024 (issue #624) and is the key change. All requested AX functions are confirmed present as free functions: AXUIElementCreateApplication, AXUIElementCopyAttributeValue, AXUIElementCopyAttributeNames, AXUIElementPerformAction, AXUIElementCopyActionNames, AXObserverCreate, AXObserverAddNotification, AXObserverGetRunLoopSource, AXIsProcessTrusted, AXIsProcessTrustedWithOptions, AXValueGetValue, AXValueCreate. Subclassing NSView, implementing delegates, and target-action/event methods are all supported from Rust via objc2's define_class! macro (with #[thread_kind = MainThreadOnly] for UI classes). No private APIs are required for the binding layer itself. The whole set is version-aligned (objc2 0.6.x core, 0.3.2 framework crates) so there are no dependency conflicts. Note: the AX bindings are auto-generated, unsafe FFI — ergonomics (CFType retain/release, AXError handling, the C observer callback trampoline) are hand-rolled on top of objc2-core-foundation, but that is normal FFI work, not a blocker.

**Key APIs**
- `objc2 define_class! macro` (objc2 0.6.4) — Subclass NSView/NSPanel/NSObject from Rust; declare ivars via #[ivars=...]; implement delegate protocols (unsafe impl NSApplicationDelegate); define target-action selectors and event handlers (mouseDown:, etc.) via #[unsafe(method(sel:))]; use #[thread_kind = MainThreadOnly] for main-thread UI/delegate classes
- `objc2 msg_send! / extern_methods! / extern_protocol!` (objc2 0.6.4) — Call ObjC methods, expose a safe Rust interface to defined methods, and declare protocols to implement
- `NSApplication / MainThreadMarker` (objc2-app-kit 0.3.2) — Run the AppKit main loop from Rust main(): NSApplication::sharedApplication(mtm) + run(); set activation policy (Accessory) for a menu-bar/overlay agent
- `NSPanel / NSWindowStyleMask::NonactivatingPanel / NSWindowLevel` (objc2-app-kit 0.3.2) — Non-activating floating overlay that does not steal focus; window level to float above app windows / over the system menu bar
- `NSVisualEffectView` (objc2-app-kit 0.3.2) — Acrylic/frosted-glass blur backing for the mirrored bar
- `AXUIElementCreateApplication / AXUIElementCopyAttributeValue / AXUIElementCopyAttributeNames` (objc2-application-services 0.3.2) — Read the target app's AX tree (AXMenuBar, menu items, titles) — free unsafe fns
- `AXUIElementPerformAction / AXUIElementCopyActionNames` (objc2-application-services 0.3.2) — Trigger the real menu action (AXPress) in the target app when a mirrored item is clicked, including submenus
- `AXObserverCreate / AXObserverAddNotification / AXObserverGetRunLoopSource` (objc2-application-services 0.3.2) — Observe AX notifications (focus/menu changes) via a C callback trampoline wired into the CFRunLoop
- `AXIsProcessTrusted / AXIsProcessTrustedWithOptions` (objc2-application-services 0.3.2) — Check/prompt for Accessibility (TCC) permission — required for any AX reads/actions
- `AXValueGetValue / AXValueCreate / AXValueType` (objc2-application-services 0.3.2) — Decode AX geometry values (CGPoint/CGSize/CGRect) for positioning the bar above the focused window
- `CFRetained / CFArray / CFDictionary / CFRunLoop` (objc2-core-foundation 0.3.2) — Manage CoreFoundation ownership and the run-loop source used by AXObserver; AXUIElement feature pulls in CFArray/CFDictionary/CFRunLoop
- `DispatchQueue (main-queue exec)` (dispatch2 0.3.1) — Marshal AX/observer work back onto the main thread for AppKit UI updates
- `RcBlock / StackBlock` (block2 0.6.2) — Provide ObjC blocks for any AppKit callback APIs that take blocks

### Permissions, packaging, and distribution (Accessibility/TCC trust, non-sandboxed .app bundling from a Rust binary, code-signing/notarization for local use on macOS 26)

**Feasibility:** `feasible-with-caveats`

Lintel's approach is feasible on macOS 26. It needs the Accessibility (TCC) grant, which is gated by AXIsProcessTrustedWithOptions(kAXTrustedCheckOptionPrompt=true); the objc2 family fully exposes the AX read/drive APIs via the objc2-application-services crate (features AXUIElement + HIServices), so no private API is needed for the AX client role. The two hard constraints are: (1) the app must be a NON-sandboxed, code-signed .app bundle — there is no App Sandbox entitlement that grants the accessibility-client role; and (2) the grant persists across rebuilds ONLY if the app has a STABLE code-signing designated requirement (stable CFBundleIdentifier + a real, reused signing certificate). An unsigned or ad-hoc-signed binary is identified by path/cdhash in TCC.db and its cdhash changes on every rebuild, silently invalidating the grant and forcing re-authorization each build — this is the classic yabai/skhd pain point. cargo-bundle can produce the bundle and set CFBundleIdentifier/version, with LSUIElement=1 injected via osx_info_plist_exts, but it does NOT code-sign, so signing must be a separate codesign step. Notarization is NOT required for a locally built app run on the same Mac (no quarantine bit), but a stable signature IS required for TCC persistence.

**Key APIs**
- `AXIsProcessTrustedWithOptions(_ options: CFDictionary?) -> bool` (ApplicationServices/HIServices) — Checks whether the process is a trusted accessibility client; with kAXTrustedCheckOptionPrompt=true it also raises the system prompt directing the user to System Settings > Privacy & Security > Accessibility. Rust: objc2_application_services::AXIsProcessTrustedWithOptions
- `AXIsProcessTrusted() -> bool` (ApplicationServices/HIServices) — Silent (no-UI) trust check for polling grant status after the user visits System Settings. Rust: objc2_application_services::AXIsProcessTrusted
- `kAXTrustedCheckOptionPrompt` (ApplicationServices/HIServices) — CFString key placed in the options dict with a CFBoolean value to request the prompt. Rust: static objc2_application_services::kAXTrustedCheckOptionPrompt
- `AXUIElementCreateApplication / AXUIElementCopyAttributeValue / AXUIElementPerformAction / AXObserverCreate` (ApplicationServices/HIServices) — The AX read/drive APIs that actually require the Accessibility grant (read the target app's menu tree, trigger AXPress on mirrored items, observe focus changes). All confirmed exposed in objc2_application_services.
- `LSUIElement` (CoreFoundation/Info.plist) — Set to 1 (true) to make Lintel an agent (no Dock icon, no default menu bar) — appropriate for a floating overlay utility.
- `CFBundleIdentifier / CFBundleVersion / CFBundleShortVersionString / LSMinimumSystemVersion` (CoreFoundation/Info.plist) — Required bundle metadata; CFBundleIdentifier must be STABLE because TCC keys the accessibility grant to it plus the code signature.
- `codesign --sign / --deep --options runtime` (Security (command-line)) — Signs the .app with a stable identity so the TCC grant survives rebuilds; --options runtime enables hardened runtime (needed only if you later notarize).
- `tccutil reset Accessibility <bundle-id>` (TCC (command-line)) — Clears a stale/broken accessibility grant during development when the csreq no longer matches.

**Rust mapping:** Use the objc2-application-services crate (part of the objc2 family, same release train as objc2-foundation/objc2-app-kit already in the plan) for the entire AX surface — enable the AXUIElement and HIServices cargo features. Confirmed exported symbols on docs.rs latest: AXIsProcessTrustedWithOptions, AXIsProcessTrusted, kAXTrustedCheckOptionPrompt, AXUIElementCreateApplication, AXUIElementCopyAttributeValue(s), AXUIElementCopyAttributeNames, AXUIElementCopyActionNames, AXUIElementPerformAction, AXUIElementSetAttributeValue, AXObserverCreate/AddNotification/GetRunLoopSource, AXError, AXValue. This avoids the older third-party `accessibility`/`accessibility-sys` crates and keeps one binding family. For packaging: cargo-bundle (`[package.metadata.bundle]` with `identifier`, `version`, and `osx_info_plist_exts = ["extra.plist"]` to inject LSUIElement and LSMinimumSystemVersion) produces the .app; note it is 'early alpha' and does NOT code-sign — run a separate `codesign` step (or a small build script / Makefile) after bundling. Alternative: hand-assemble the .app tree (Contents/MacOS, Contents/Info.plist) with a build script, optionally using the embed_plist crate to embed Info.plist into a plain binary, but a real .app directory is preferable so TCC lists it cleanly by bundle ID.

**Confirmed behaviors**
- AXIsProcessTrustedWithOptions with kAXTrustedCheckOptionPrompt=true is the documented, non-private way to check and prompt for the Accessibility grant (Apple Developer docs).
- The objc2-application-services crate exposes the full AX client API (trust check + create/read/perform/observe) — verified against docs.rs symbol listing — so the AX-client role needs no private API.
- App Sandbox has no standard entitlement for the accessibility-client role; a broad AX client must be distributed non-sandboxed (outside the Mac App Store).
- TCC.db stores per-grant fields including client (bundle ID or absolute path), client_type (0=bundle ID, 1=absolute path — the latter is 'common for debug tools or Xcode projects'), and a csreq code-signing-requirement binary blob (Huntress TCC teardown), confirming grants are tied to app identity + signature.
- cargo-bundle maps Cargo `identifier`->CFBundleIdentifier and `version`->bundle version, supports appending arbitrary Info.plist keys (e.g. LSUIElement) via osx_info_plist_exts, and does not perform code signing.
- tccutil reset Accessibility resets accessibility grants; the grant showing as 'checked' in System Settings can still be non-functional if the underlying record is stale/mismatched.

**Risks**
- (high) Grant is invalidated on every rebuild if the app is unsigned or ad-hoc-signed (cdhash changes each build; TCC identifies it by path/cdhash, not a stable designated requirement). This makes iterative development miserable and would make the shipped app re-prompt after each update. — *mitigation:* Sign every build with a STABLE identity (a self-signed 'Developer ID Application'-style cert kept in the login keychain, or a real Developer ID). Keep CFBundleIdentifier constant. This gives a constant designated requirement so the TCC grant persists across rebuilds. Automate `codesign` in the build. During dev, `tccutil reset Accessibility <id>` clears stale entries.
- (medium) App Sandbox cannot host the AX-client role; if the app is ever sandboxed (e.g. to consider Mac App Store) the feature breaks — there is no approved entitlement for arbitrary accessibility control. — *mitigation:* Ship non-sandboxed, distributed outside the Mac App Store (direct download / personal build). Do not add com.apple.security.app-sandbox. This matches the plan's 'non-sandboxed' requirement.
- (medium) cargo-bundle is early-alpha and does not code-sign; relying on it alone yields an unsigned bundle that will hit the rebuild-invalidation problem and may need extra Info.plist keys it doesn't emit. — *mitigation:* Treat bundling and signing as two steps: cargo-bundle (or a hand-rolled build script) for the .app + Info.plist, then a scripted codesign --deep --options runtime pass. Verify the plist actually contains LSUIElement, CFBundleIdentifier, and LSMinimumSystemVersion after bundling.
- (low) Notarization confusion: assuming notarization is needed for personal use, or assuming Gatekeeper will block a locally built app. — *mitigation:* A locally built .app run on the same Mac has no com.apple.quarantine attribute, so Gatekeeper does not block it and notarization is NOT required for personal/local use. Notarization (+ hardened runtime) is only needed if the .app is downloaded/distributed to other machines. Stable signing is still needed regardless, for TCC persistence.
- (low) macOS 26 System Settings UI has reported quirks showing binary/CLI accessibility clients inconsistently (yabai issue on 26.1), which can confuse the grant flow for a path-identified client. — *mitigation:* Ship as a proper .app bundle (bundle-ID-identified, client_type=0), not a bare binary; this makes the entry render reliably in Privacy & Security > Accessibility and behave as a stable client.

**Critical claim:** The riskiest assumption is that the Accessibility grant will persist across development rebuilds and app updates. It only does so if the app carries a STABLE code-signing designated requirement (stable CFBundleIdentifier + a reused signing certificate). An unsigned/ad-hoc build is keyed by path/cdhash and re-invalidates on every rebuild — so 'sign every build with the same real certificate' is a non-negotiable prerequisite, not an optional polish step, and must be wired into the build from day one.

### Geometry and multi-display/ultrawide placement (converting AX window frames to Cocoa screen coordinates, NSScreen enumeration, which-screen detection, placement algorithm, and the "window touches top but not fullscreen" case)

**Feasibility:** `feasible`

Positioning a floating NSPanel exactly above a focused window's top edge is fully feasible with public APIs. AX window geometry (kAXPositionAttribute/kAXSizeAttribute) is reported in the Core Graphics GLOBAL display coordinate space: origin at the TOP-LEFT of the PRIMARY display, y increasing DOWNWARD, in points (not pixels). NSScreen/Cocoa uses the BOTTOM-LEFT of the primary display as origin with y increasing UPWARD, also in points. Because both spaces are anchored to the SAME primary display and both are in points, a single y-flip using the primary screen's height converts between them regardless of how many displays exist or their individual backingScaleFactor. Formula: cocoa_x = ax_x; cocoa_top_y = primaryHeight - ax_y; the window's Cocoa origin (bottom-left) is cocoa_y = primaryHeight - (ax_y + ax_h). To pin the bar above the window, set panel.origin = (ax_x, primaryHeight - ax_y) so the panel's bottom edge sits on the window's top edge, then clamp. All required NSScreen accessors exist in objc2-app-kit. The riskiest piece is the top-collision/menu-bar-overlay behavior, which relies on window-level ordering (feasible) rather than moving the real menu bar (infeasible without private API).

**Key APIs**
- `NSScreen::screens(mtm) -> Retained<NSArray<NSScreen>>` (objc2-app-kit (feature "NSScreen")) — Enumerate all attached displays; index 0 is the primary/origin screen (its frame.origin == (0,0)).
- `NSScreen::mainScreen(mtm) -> Option<Retained<NSScreen>>` (objc2-app-kit) — The screen with the key/focused window; NOT necessarily the primary/origin screen. Do not use it as the flip reference.
- `NSScreen::frame(&self) -> NSRect` (objc2-app-kit) — Full display rect in the global Cocoa (bottom-left) space; screens[0].frame.size.height is the flip reference (primaryHeight).
- `NSScreen::visibleFrame(&self) -> NSRect` (objc2-app-kit) — Display rect minus the menu bar and Dock; visibleFrame.maxY is the highest y a bar can occupy without overlapping the menu bar on the menu-bar-hosting screen.
- `NSScreen::backingScaleFactor(&self) -> CGFloat` (objc2-app-kit (needs objc2-core-foundation)) — Per-display HiDPI scale for choosing bar thickness/crispness; NOT needed for the coordinate flip (points are unified across displays).
- `NSScreen safeAreaInsets / auxiliaryTopLeftArea / auxiliaryTopRightArea` (objc2-app-kit (macOS 12+)) — Notch geometry; needed if the bar is drawn over the menu bar region on a notched built-in display.
- `AXUIElementCopyAttributeValue with kAXPositionAttribute / kAXSizeAttribute` (objc2-application-services / ApplicationServices (HIServices)) — Read the focused window's frame; values are AXValue of kAXValueTypeCGPoint / kAXValueTypeCGSize in the global CG top-left space.
- `AXValueGetValue(value, kAXValueTypeCGPoint, &pt) / kAXValueTypeCGSize` (ApplicationServices) — Unbox the AXValue into CGPoint/CGSize before conversion.
- `NSRect / NSPoint / NSSize (== CGRect/CGPoint/CGSize)` (objc2-foundation / objc2-core-foundation) — Geometry types shared between AX (CGRect) and AppKit (NSRect) — same layout, so no struct conversion is needed, only the y-flip.
- `NSWindow::setFrameOrigin / setFrame_display` (objc2-app-kit) — Place the panel at the computed Cocoa origin.
- `NSWindow::setLevel(NSWindowLevel) with NSStatusWindowLevel(25)/NSPopUpMenuWindowLevel(101) vs NSMainMenuWindowLevel(24)` (objc2-app-kit) — To draw the acrylic bar OVER the system menu bar in the top-collision case; any level > 24 renders above the menu bar.
- `CGDisplayBounds(CGDirectDisplayID)` (objc2-core-graphics) — Cross-check: returns display bounds in the same top-left global space AX uses; useful to map an AX point to a CGDirectDisplayID and validate the flip.

**Rust mapping:** All bindings are public/safe in objc2-app-kit. Coordinate flip in Rust: let mtm = MainThreadMarker::new().unwrap(); let screens = NSScreen::screens(mtm); let primary_h = screens.objectAtIndex(0).frame().size.height; // primary defines origin. AX read yields CGPoint{ax_x,ax_y}, CGSize{w,h}. Compute: let origin = NSPoint{ x: ax_x, y: primary_h - ax_y }; // panel bottom == window top. NSRect and CGRect are the same repr (objc2-foundation re-exports objc2-core-foundation CGRect as NSRect), so AXValueGetValue can fill a CGRect that is directly usable by AppKit — no field-by-field conversion. Which-screen detection: iterate screens, compute intersection area of the window's Cocoa rect with each screen.frame (NSIntersectionRect / manual max(0,...) products), pick max area; this matches how macOS assigns window ownership and beats center-point containment for windows straddling displays. Ultrawide: nothing special is required — the far-from-menu-bar window simply has a large positive x; the bar anchors to that x (clamped to the owning screen), which is exactly the product goal of bringing the menu to the window. ### Prior art that proves or limits feasibility (existing macOS apps + OSS that mirror/relocate menus, read/trigger other apps' menus via AX, overlay the system menu bar, and Rust AX projects)

**Feasibility:** `feasible-with-caveats`

The core of Lintel is proven shipping reality: Many Tricks' Menuwhere has commercially read the frontmost app's ENTIRE menu tree over the Accessibility API and triggered items (including submenus, and even background apps' menus) since ~2017, and the open-source right-click-menubar (Swift, MIT) reproduces the exact AX code path end to end. So "read another app's menu bar + trigger its actions via AX" is unambiguously feasible. What NO prior art demonstrates as a single package is Lintel's specific combination: a PERSISTENT, horizontal, non-activating, acrylic floating bar (rather than a transient popup) that mirrors the menu live and can be drawn OVER the system menu bar. Menu-bar-region overlay is grounded by managers like Ice/Bartender/Hidden Bar (they position bars/windows in the menu-bar zone) plus the NSWindow level model (a window at level >24 draws above the level-24 main menu bar). The dominant limitation surfaced by prior art is LAZY SUBMENU LOADING: context/selection-dependent submenus are frequently empty over AX until the real menu is opened — Menuwhere ships an explicit "Preload menus" workaround for exactly this, which is the clearest proof the problem is real and solvable. A secondary warning: macOS 26 Tahoe / Liquid Glass changed menu-bar behavior and broke menu-bar managers (Ice/Bartender needed rework), so menu-bar-region tricks are OS-version fragile.

**Key APIs**
- `AXUIElementCreateApplication(pid)` (ApplicationServices/HIServices (C)) — Get the AX root for the frontmost app (from NSWorkspace.frontmostApplication.processIdentifier); starting point for reading its menu bar. Proven in right-click-menubar.
- `AXUIElementCopyAttributeValue + kAXMenuBarAttribute` (ApplicationServices) — Fetch the app's AXMenuBar element from the app root. Exact call used in right-click-menubar.
- `kAXChildrenAttribute / kAXRoleAttribute (AXMenuBarItem, AXMenu, AXMenuItem)` (ApplicationServices) — Recursively walk the menu tree; role distinguishes top-level bar items, submenus, and leaf items.
- `kAXTitleAttribute, kAXEnabledAttribute, AXMenuItemCmdChar, AXMenuItemCmdModifiers` (ApplicationServices) — Read each item's label, enabled state, and keyboard shortcut for faithful mirroring (right-click-menubar reads all of these).
- `AXUIElementPerformAction + kAXPressAction` (ApplicationServices) — Trigger the real menu action in the target app on a mirrored item click — the core interactivity mechanism, proven.
- `AXIsProcessTrusted() / AXIsProcessTrustedWithOptions` (ApplicationServices) — Gate: Accessibility permission is mandatory to read/drive other apps' menus (both Menuwhere and right-click-menubar require it).
- `NSWindow.level (mainMenu=24, statusBar=25, popUpMenu/screenSaver=101)` (AppKit) — Set the acrylic bar above the level-24 system menu bar to draw OVER it; grounds the 'cover the menu bar' requirement.
- `NSMenu.popUp(positioning:at:in:) + NSMenuItem.representedObject/validateMenuItem` (AppKit) — Prior-art shortcut (right-click-menubar/Menuwhere): rebuild a NATIVE NSMenu from AX data so submenu tracking/keyboard nav are native, and only route the final click to AXPress. Note: this is a transient popup, not Lintel's persistent bar.
- `axuielement crate v0.9.1` (Rust (crates.io/docs.rs)) — Safe Rust bindings: 'drive other apps' UIs — read attributes, perform actions, subscribe to notifications' — the highest-level Rust option for the AX work.
- `accessibility-sys v0.2.0 (+ accessibility v0.2.0)` (Rust (eiz/accessibility)) — Complete low-level FFI to the AX C functions/constants above; the reliable Rust binding path to pair with the objc2 UI layer (AX is a C API, not part of objc2).
- `macos-accessibility-client v0.0.2` (Rust) — Wraps the AXIsProcessTrusted permission check/prompt from Rust.

**Rust mapping:** objc2 / objc2-app-kit / objc2-foundation cover the UI side (NSWindow/NSPanel, NSVisualEffectView for acrylic, NSWorkspace for frontmost-app + focus notifications), but they do NOT include the Accessibility API — AX is a plain C API in ApplicationServices/HIServices. Bridge it with a -sys crate: use axuielement 0.9.1 (safe, actively versioned to 2026) for the ergonomic path, or accessibility-sys 0.2.0 (described as 'complete') for direct FFI mirroring the exact Swift calls in right-click-menubar (AXUIElementCreateApplication, AXUIElementCopyAttributeValue(kAXMenuBar/kAXChildren/kAXTitle/kAXRole/kAXEnabled/AXMenuItemCmdChar/AXMenuItemCmdModifiers), AXUIElementPerformAction(kAXPress)). These interoperate cleanly because AX types are CoreFoundation CFTypeRefs, shared with objc2-core-foundation. macos-accessibility-client covers the AXIsProcessTrusted permission prompt. Architectural option worth stealing from prior art: reconstruct a native NSMenu (via objc2-app-kit) for submenu tracking and only send the terminal click through AXPress — but that yields a popup, whereas Lintel wants a persistent bar, so the bar's top-level items likely need custom drawing with per-item lazy AX population on hover.
